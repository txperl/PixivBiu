# AGENTS.md

## Project Overview

[PixivBiu](https://github.com/txperl/PixivBiu) is a Pixiv artwork browsing, searching, and downloading tool. **v3 is its next major version**, adopting Go as the implementation language and evolving the project structure to a decoupled backend + frontend monorepo.

The current state is a **framework skeleton only** — no Pixiv business logic has been implemented yet. Subsequent iterations will add authentication, API proxying, downloading, caching, etc.

## Architecture Overview

| Component | Tech Stack | Default Port | Description |
|-----------|-----------|--------------|-------------|
| **Backend** | Go + chi + oapi-codegen | 8080 | REST API server. OpenAPI-first: routes and types generated from `api/openapi.yaml`. |
| **Frontend** | TBD | TBD | Placeholder directory. Tech stack undecided. Talks to backend via `/api/v1/*`. |

## Tech Stack

### Backend (Go)

- **Language**: Go 1.26
- **Router**: [chi v5](https://github.com/go-chi/chi) — lightweight, idiomatic HTTP router
- **API codegen**: [oapi-codegen v2](https://github.com/oapi-codegen/oapi-codegen) — generates types + chi server interface from OpenAPI 3 spec. Installed as a `go tool` (Go 1.24+ tool directive).
- **Config**: [koanf v2](https://github.com/knadh/koanf) — providers for confmap (defaults), file+yaml, env
- **Logging**: standard library `log/slog` — structured, text/JSON output
- **Decode hooks**: `github.com/go-viper/mapstructure/v2` (for `time.Duration` parsing)

### Frontend

Undecided. See `frontend/README.md`.

## Development Guide

### Prerequisites

1. Go 1.26+
2. `make`
3. One-time setup:
   ```bash
   go mod tidy
   go get -tool github.com/oapi-codegen/oapi-codegen/v2/cmd/oapi-codegen@latest
   cp config.example.yaml config.yaml
   ```

## Key Development Rules

### OpenAPI Workflow (most important)

This is an **OpenAPI-first** project. The spec at `api/openapi.yaml` is the **single source of truth** for all HTTP routes, request/response schemas, and model types. The development loop:

1. **Edit `api/openapi.yaml`** to add or change routes, parameters, or schemas.
2. **Run `make gen`** to regenerate `internal/api/server.gen.go`. This file contains:
   - Go types for all schemas (e.g., `HealthStatus`).
   - The `ServerInterface` Go interface that your handler must implement.
   - Chi routing glue (`HandlerFromMux`, `HandlerFromMuxWithBaseURL`, etc.).
3. **Update `internal/api/handler.go`** — add methods to `APIHandler` to satisfy new `ServerInterface` methods. The compiler will fail until every interface method is implemented (this is intentional — it's how we ensure the handler stays in sync with the spec).
4. **Run `make run`** to verify.

#### Rules

- ⚠️ **DO NOT modify `*.gen.go` files manually.** They are regenerated on every `make gen` and your edits will be lost. If the output is wrong, fix `api/openapi.yaml` or `api/cfg.yaml` instead.
- **DO NOT hand-write route registrations** in `internal/server/server.go`. Routes come from `api.HandlerFromMuxWithBaseURL(handler, router, "/api/v1")`. Adding new endpoints means editing the spec, not the router.
- **DO NOT bypass the `ServerInterface`.** Every endpoint must go through the generated interface so spec and implementation stay coupled.
- **Keep `operationId` unique and camelCase** in the spec — it becomes the Go method name on `ServerInterface` (e.g., `getHealth` → `GetHealth`).
- **The `servers:` block uses `/api/v1` as base URL.** Paths in the spec are written without the `/api/v1` prefix. The prefix is applied at mount time via `HandlerFromMuxWithBaseURL`.

### Naming Convention: `APIHandler` vs `Handler`

The generated `internal/api/server.gen.go` exports a function named `Handler(si ServerInterface) http.Handler`. To avoid a name collision, our handler struct is named **`APIHandler`**, not `Handler`:

```go
type APIHandler struct { logger *slog.Logger }
func NewHandler(logger *slog.Logger) *APIHandler { ... }
```

When adding new handler files inside `internal/api/`, attach methods to `APIHandler`. Do not introduce another type named `Handler` in this package.

### Handler Implementation

- All handler methods live in `internal/api/handler.go` (or sibling files in the same package, grouped by domain once the surface grows).
- Use the `writeJSON(w, status, v)` helper already defined in `handler.go` for JSON responses.
- Access the logger via `h.logger`. Do not use `slog.Default()` inside handlers — the injected logger may carry request-scoped attributes in the future.
- Return errors through proper HTTP status codes and JSON error bodies; do not `panic` (the `Recoverer` middleware will catch panics but that should be a safety net, not a flow-control tool).

### Configuration

Precedence (low → high): **defaults → `config.yaml` → environment variables**.

- Defaults live in `internal/config/config.go` (`defaults()` function). Add new keys there as well as updating the Go struct.
- The YAML schema is in `config.example.yaml`. Keep it in sync with the Go struct and the defaults function.
- Environment variable convention: prefix `PIXIVBIU_`, underscores map to dots. For example:
  - `PIXIVBIU_SERVER_PORT=9090` → `server.port`
  - `PIXIVBIU_SERVER_TIMEOUTS_SHUTDOWN=20s` → `server.timeouts.shutdown`
  - `PIXIVBIU_LOG_LEVEL=debug` → `log.level`
- YAML keys are **all lowercase** (no camelCase, no underscores) so the env-var naming maps cleanly. When adding multi-word concepts, group them under a nested section (see `server.timeouts.{read,write,shutdown}`) instead of using underscores in keys.
- `time.Duration` fields accept Go duration strings (`"15s"`, `"1m30s"`). The mapstructure decode hook handles the conversion.

### Logging

- Use `log/slog` — no third-party logger.
- Level values (case-insensitive): `debug`, `info`, `warn`, `error`.
- Format values: `text` (default), `json`.
- Inside handlers, use the `h.logger` injected at construction. Outside of request paths, use `slog.Default()` — it's set globally in `main.run()` after config load.
- Request-level attributes (method, path, status, duration, request_id, remote) are emitted by the `requestLogger` middleware in `internal/server/server.go`. Do not duplicate those fields in handler logs.

### Middleware

Middleware order in `internal/server/server.go` is significant:

```
RequestID → RealIP → Recoverer → requestLogger → (generated routes)
```

- Add new middleware **after** `Recoverer` so panics in new middleware are caught.
- Add new middleware **before** `requestLogger` only if you need the log line to reflect its effect.
- Do not reorder existing middleware without understanding the impact on request IDs and panic recovery.

### Graceful Shutdown

`cmd/server/main.go` listens for `SIGINT` / `SIGTERM` and calls `srv.Shutdown(ctx)` with `cfg.Server.Timeouts.Shutdown` as the deadline. When adding long-lived background workers (queues, schedulers, connection pools), plumb their shutdown into the same `ctx` so they stop alongside the HTTP server.

## Configuration Reference

The canonical template is `config.example.yaml`. Current sections:

- `server.host` — bind address (default `0.0.0.0`)
- `server.port` — TCP port (default `8080`)
- `server.timeouts.read` — `http.Server.ReadTimeout` (default `15s`)
- `server.timeouts.write` — `http.Server.WriteTimeout` (default `15s`)
- `server.timeouts.shutdown` — graceful shutdown deadline (default `10s`)
- `log.level` — `debug` | `info` | `warn` | `error` (default `info`)
- `log.format` — `text` | `json` (default `text`)

Local config file path can be overridden with `-config <path>`. If the file does not exist, defaults + env are used (no error).

## Out of Scope (Current Phase)

The following are intentionally **not** implemented yet — do not add them without explicit instruction:

- Pixiv OAuth / token refresh / session management
- Downloader, cache, proxy, rate-limiter
- Authentication / authorization middleware
- Metrics, tracing, health probes beyond `/health`
- Database, ORM, migrations
- Docker / CI pipelines
- Frontend implementation

When adding features, prefer **extending the OpenAPI spec first**, then generating and implementing — not inventing ad-hoc routes or helper packages.
